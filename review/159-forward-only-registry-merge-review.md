# Review 159 - forward-only registry merge

Reviewed commit `144b72c5` (`forge-storage/schema: forward-only per-collection registry merge on migration import (DL-13 review 147)`).

## Finding

- **P1 - Non-superset carried collections are skipped even when they are concurrent additive evolution.** `sync_collection_forward_only` treats every carried collection that is not a forward-compatible superset of the local entry as stale and returns `Ok(false)` (`forge/crates/schema/src/registry.rs:363-365`). The import path still imports the chunk and advances the carried `schema_version` before calling this merge (`forge/crates/storage/src/crdt_write/remote.rs:280-302`). That is safe for the new stale-narrowing test, but it breaks the DL-11 shape where two offline users add different fields to the same collection and the registry must converge to the union (`prd-merged/02-data-layer-prd.md:61`). Example: receiver already has local field `f_bob_0`, then imports an authorized migration carrying remote field `f_alice_0` from the same base. Because the carried collection does not contain `f_bob_0`, the superset check fails and the remote registry entry is skipped, while the migrated records/version still land. The receiver now has data for the remote field but no registry entry for it. Please either implement a field-id union merge for additive divergence (preserve local-only and carried-only fields, require shared fields to be forward-compatible, max actor counters) or fail closed when the carried entry is neither stale-backward nor mergeable. Add a regression where local and carried collections each have a different actor-scoped field and the post-import registry contains both.
