## Review: 7d3a0b41 remaining audit producers

Found one correctness bug in the new runtime-derived network/secret audit rows.

- **P1 - Denied `net.fetch` attempts are persisted as `allow` network/secret rows.** The runtime recorder intentionally stores policy-denied host calls as `{"denied": <CoreError>}` entries in the run trace (`forge/crates/runtime/src/recorder.rs:313`), and the existing net host tests show a non-allowlisted fetch records a `net.fetch` call whose response has `denied` and never reaches the bridge (`forge/crates/runtime/src/host/net.rs:400`). The new `persist_run_egress_audit` walks every recorded `net.fetch` without checking that denial shape, defaults missing `status` to `0`, and then appends `secret.use` and `network.egress` rows with `decision = "allow"` / reason `"network policy allowed request"` (`forge/crates/core/src/commands/runtime_run.rs:298`, `forge/crates/core/src/commands/runtime_run.rs:325`, `forge/crates/core/src/commands/runtime_run.rs:367`). That makes an attempted forbidden egress or disallowed secret header look approved in the durable audit log. Please detect `call.response.denied` and either emit deny-shaped audit rows with the denial reason, or skip the allow producers and add a separate denial producer row; add a live test for a denied `ctx.net.fetch` with a `secret_ref` header asserting no `allow` `network.egress`/`secret.use` rows are written.
