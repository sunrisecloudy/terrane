## Review: 85023b99 migration registry merge

Found one follow-up risk in the new per-collection registry merge.

- **P1 - Out-of-order older migration chunks can replace a newer collection registry.** The fix now calls `evolve_registry_collection_tx` whenever a migration chunk carries `registry_collection`, even when the receiver's workspace `schema_version` is already at or beyond the chunk target (`forge/crates/storage/src/crdt_write/remote.rs:227`). That is needed for the same-version/unrelated-work case, but `evolve_registry_collection_tx` then calls `SchemaRegistry::sync_collection` unconditionally (`forge/crates/storage/src/crdt_write/remote.rs:306`), and `sync_collection` blindly replaces the local collection (`forge/crates/schema/src/registry.rs:308`). If a peer has already evolved that same collection past the incoming migration target, a delayed older chunk can roll the registry backward while the global version stays newer. Please make this merge forward-only per collection (or verify the carried definition is a compatible superset of the local one before replacing) and add an out-of-order migration test where `expenses` is already ahead before an older migration chunk arrives.
