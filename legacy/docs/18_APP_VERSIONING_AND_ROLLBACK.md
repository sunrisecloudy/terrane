# App Versioning and Rollback

## 1. Immutable versions

Every installed app version is immutable. Installing a new version creates a new version record; it must not overwrite the old one.

```text
notes-lite
  versions/
    0.1.0+001/
    0.1.0+002/
    0.2.0+001/
  active -> 0.2.0+001
```

Use `schemas/app-version-record.schema.json` for installed-version metadata.

## 2. Version identifiers

Use two concepts:

- `manifest.version`: semantic version generated by the app author/AI.
- `installId`: unique installed version id generated by the platform.

Example:

```json
{
  "appId": "notes-lite",
  "appVersion": "0.1.0",
  "installId": "notes-lite-0.1.0-20260528T120000Z-a1b2c3",
  "status": "enabled"
}
```

## 3. Statuses

| Status | Meaning |
|---|---|
| `installed` | Stored but not active |
| `enabled` | Active version users can open |
| `disabled` | Kept but hidden |
| `quarantined` | Failed audit/test/runtime check |
| `rolled-back` | Replaced by an earlier version |
| `uninstalled` | Removed from launcher, metadata may remain for audit |

## 4. Rollback rules

Rollback must:

1. Disable the current active version.
2. Restore the previous enabled version pointer.
3. Preserve storage unless the rollback explicitly restores a snapshot.
4. Record a rollback event in the app version history.
5. Run smoke tests after rollback.

If the app's data has already migrated to a newer `dataVersion`, rollback must either:

- use a compatible older app version that supports the current data version;
- run a declared down migration if available;
- restore a pre-migration snapshot;
- refuse rollback and explain why.

## 5. Quarantine triggers

Automatically quarantine an installed version if:

- signature verification fails;
- content hash changes;
- manifest validation fails;
- it exceeds resource budgets repeatedly;
- it calls forbidden bridge methods;
- micro-tests fail after install;
- runtime crashes or throws fatal errors during mount.

## 6. Codex tools

The control-plane/MCP layer must expose:

- `platform.list_webapp_versions`;
- `platform.rollback_webapp`;
- `platform.quarantine_webapp`;
- `platform.install_report`.

Codex must prefer rollback over ad-hoc patching when a newly generated version breaks a previously working app.

## Database persistence mapping

Versioning and rollback are database-backed:

```text
apps.active_install_id
app_versions.install_id
app_versions.version
app_versions.manifest_json
app_versions.content_hash
app_files.install_id/path
app_permissions.install_id/permission
app_installations.action=rollback
```

Rollback updates only the active install pointer and app installation history unless an explicit data rollback is requested from a pre-migration snapshot.
