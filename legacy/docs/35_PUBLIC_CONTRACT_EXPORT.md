# Public Contract Export

## 1. Purpose

This document defines the machine-readable Terrane public contract consumed by private products such as Terrane Premium.

The public contract is the release artifact that lets another repo pin Terrane runtime behavior without copying implementation details by hand. It lists the public docs, schemas, fixtures, tools, bridge methods, and release commands that define generated-app-visible behavior.

Private products may depend on this contract. They must not define hidden generated-app runtime semantics outside it.

## 2. Contract Artifact

The release artifact is:

```text
artifacts/public-contract.json
```

It is generated by:

```text
node --no-warnings tools/export-public-contract.mjs --out artifacts/public-contract.json
```

`tools/package-release.mjs` also writes this file during normal release packaging and records its hash in `artifacts/release-manifest.json`.

## 3. Contract Contents

The contract includes:

- contract identity and schema version;
- provenance for the Terrane source commit, optional release tag, generator, and release-manifest artifact id;
- public platform baseline;
- generated-app boundary rules;
- bridge method list;
- notebook bridge method list;
- runtime event surface;
- syncable record kinds that private SaaS products may mirror;
- non-syncable local record kinds that must not sync by default;
- sync record mappings to public local tables and private adapter resources;
- non-syncable record mappings to local tables and exclusion reasons;
- public docs with SHA-256 hashes;
- Forge contract files with SHA-256 hashes;
- Forge fixtures and example applets with SHA-256 hashes;
- public tools that private CI may call or inspect;
- expected CI commands.

## 4. Stability Rules

Generated-app-visible behavior is contract behavior.

Examples:

- bridge method request and response shapes;
- package file rules;
- manifest fields;
- package signing and canonicalization;
- backup/export/import shape;
- notebook CRDT bridge behavior;
- runtime capability shape;
- network policy semantics.

When one of these changes:

1. update the public doc or schema;
2. update fixtures;
3. update public conformance tests;
4. regenerate the public contract artifact;
5. let private downstream CI fail until it adapts or intentionally pins the prior contract.

## 5. Non-Goals

- The public contract does not expose private SaaS auth, billing, admin, abuse, or infrastructure logic.
- The public contract does not make the OSS local server a production multi-tenant SaaS backend.
- The public contract does not grant generated apps direct SaaS API access.
- The public contract does not include secrets, customer data, local control tokens, or private signing keys.

## 6. Private SaaS Consumption

A private SaaS repo should:

- pin a Terrane commit, tag, or release artifact;
- read `public-contract.json`;
- verify its schema and hashes;
- run `node --no-warnings tools/verify-public-contract.mjs --contract artifacts/public-contract.json --root .` against the matching Terrane checkout;
- verify the source commit or release tag provenance matches the pinned Terrane source;
- run public package/signing/backup/bridge/CRDT conformance tests relevant to the private features it consumes;
- fail release if the pinned public contract and private adapters drift.

The private repo should keep one adapter layer for the public/private boundary. It should not reach into public Terrane internals from many services.

The conformance commands listed in the contract require a Terrane source checkout at the same commit as `provenance.sourceCommit`. The JSON artifact alone is enough to inspect the contract and verify file hashes, but it is not enough to run source-tree tests unless a future release also ships a separate conformance bundle.

## 7. Acceptance

The public contract export is complete when:

- `forge/contracts/public-contract.schema.json` exists;
- `tools/export-public-contract.mjs` generates deterministic JSON;
- `tools/verify-public-contract.mjs` verifies schema, provenance, and all recorded file hashes;
- `tools/package-release.mjs` includes `public-contract.json`;
- `tools/check-repo.mjs` validates the generated contract against its schema;
- reference-host tests verify the artifact and release manifest entry;
- this document lists the downstream consumption rules.
