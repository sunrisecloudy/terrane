# Review 086 - commit 6ec33540

## Findings

- **[P2] Reject unknown signed policy fields instead of normalizing them away.** The signed package manifest is kept as raw JSON and hashed whole, but the new bind narrows it through today-only shapes before comparing: `networkPolicy.allow[]` is deserialized as `forge_domain::NetRule` (`forge/crates/core/src/workspace.rs:2361-2377`), whose type explicitly tolerates unknown constraint fields for forward compatibility (`forge/crates/domain/src/manifest.rs:143-147`), and `resourceBudget` checks only six known keys while silently ignoring any other signed budget key (`forge/crates/core/src/workspace.rs:2239-2255`). A package can therefore carry a signed future/tighter policy field, e.g. a new network constraint or `resourceBudget.network_bytes`/`output_bytes`, and this core will install it as `Signed` while not enforcing that signed constraint. That conflicts with the v1 compatibility model where packages declare required features and clients refuse unsupported ones (`prd-merged/08-marketplace-prd.md:24`). Safe M0a fix: fail closed for signed packages that contain unknown keys in `capabilities`, `networkPolicy.allow[]`, or `resourceBudget` unless a negotiated `required_features` path explicitly says this core supports them.

- **[P3] Add a net regression that keeps the signed route identical.** The new test `a_signature_cannot_loosen_a_net_cap_or_add_a_secret_header` adds a net rule to a signed package whose `networkPolicy.allow` is empty (`forge/crates/core/tests/spine.rs:2316-2359`; current signing fixtures all have empty allowlists). That test still fails if the binder regresses back to comparing only `(method, url)`, because the signed set is empty and the install set is non-empty. To pin review 083 #3, add a signed fixture with one net allow rule, then install the same method/url with a wider cap or extra `allow_secret_headers` and assert rejection.
