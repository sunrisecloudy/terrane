# Commit Review: 70f5cbff storage index lifecycle

Reviewed commit:
- `70f5cbff forge-storage: create/drop/deprecate index API + FTS sync on mutation (DL-5/DL-6)`

## Findings

1. **P1 - The DL-17 mutation path still bypasses FTS maintenance.** The spec requires any insert/update/delete to the records projection to refresh active FTS rows in the same logical transaction (`forge/spec/dynamic-indexes.md:83`). This commit adds `put_record_indexed`/`patch_record_indexed`/`delete_record_indexed` helpers (`forge/crates/storage/src/lib.rs:645`, `forge/crates/storage/src/lib.rs:662`, `forge/crates/storage/src/lib.rs:689`), but the existing applet-facing mutation methods still call the unsynced paths: `apply_mutation` uses `put_record`, `update_record`, `patch_record`, and `delete_record` (`forge/crates/storage/src/lib.rs:775`, `forge/crates/storage/src/lib.rs:793`, `forge/crates/storage/src/lib.rs:802`, `forge/crates/storage/src/lib.rs:811`, `forge/crates/storage/src/lib.rs:819`), while `transact_mutations` uses `apply_mutation_tx`, which only calls `put_record_tx` (`forge/crates/storage/src/lib.rs:834`, `forge/crates/storage/src/lib.rs:1192`, `forge/crates/storage/src/lib.rs:1210`, `forge/crates/storage/src/lib.rs:1224`, `forge/crates/storage/src/lib.rs:1240`, `forge/crates/storage/src/lib.rs:1253`). Active text-search indexes still go stale for the primary DL-17 API. Please thread `IndexManager`/registry index state through the real mutation path, or make the unsynced APIs projection-only/private so callers cannot accidentally violate DL-5.

2. **P1 - Creating both value and FTS indexes for one field overwrites metadata and orphans the previous physical index.** DL-5 distinguishes expression indexes and FTS5 shadow tables, and the canonical index definition includes `kind` (`prd-merged/02-data-layer-prd.md:50`, `forge/spec/dynamic-indexes.md:31`). `IndexManager` still keys definitions only by `(collection, field_id)` (`forge/crates/storage/src/index.rs:292`), and `register` replaces any existing entry (`forge/crates/storage/src/index.rs:310`). With the new `create_index`, creating `Value` then `Fts` builds the FTS table and overwrites the manager entry without dropping the expression index; creating `Fts` then `Value` leaves the FTS table behind but removes it from planning/sync (`forge/crates/storage/src/index.rs:352`, `forge/crates/storage/src/index.rs:364`, `forge/crates/storage/src/index.rs:370`). `drop_index`/`deprecate_index` can then only see the last registered kind (`forge/crates/storage/src/index.rs:379`, `forge/crates/storage/src/index.rs:400`). Key definitions by `(collection, field_id, kind)` or add a composite definition that can own both physical structures.

3. **P2 - Text-search queries still bypass the normal query pipeline.** This commit improves FTS freshness for the new helper methods, but `query_planned` still returns immediately on `q.text_search` (`forge/crates/storage/src/lib.rs:513`) and `run_text_search` only gathers ids/envelopes from FTS or fallback scan (`forge/crates/storage/src/lib.rs:545`, `forge/crates/storage/src/lib.rs:555`). It never applies `q.filter`, aggregate/group handling, or rank-path `limit`/`offset`; it also sorts rank/default results by id (`forge/crates/storage/src/lib.rs:569`, `forge/crates/storage/src/lib.rs:574`) after `fts_match` already ordered by rank (`forge/crates/storage/src/index.rs:559`). That still contradicts DL-15's combined filter/sort/limit/offset/aggregate/group/text-search surface (`prd-merged/02-data-layer-prd.md:68`). Add fixtures that combine `text_search` with filter, limit, and aggregate/group before considering the FTS path closed.

4. **P2 - New FTS sync repeats the unquoted stable-field-id JSON path bug.** Schema can mint field ids directly from actor ids, and actor validation only rejects empty strings (`forge/crates/schema/src/collection.rs:154`, `forge/crates/schema/src/registry.rs:200`). Storage allows `.` and `/` in ids (`forge/crates/storage/src/query.rs:240`) and the new incremental sync extracts with `$.field_ids.<id>` (`forge/crates/storage/src/index.rs:741`), just like rebuild population (`forge/crates/storage/src/index.rs:516`). A valid field id such as `f_dev.01_0` is read as nested JSON keys, so `create_index`/`sync_fts_for_record` can build and maintain an empty/wrong FTS table for schema-valid ids. Either constrain schema field ids to JSON-path-safe tokens or use quoted JSON paths consistently in query, DDL, rebuild, and incremental sync.

5. **P2 - Planner full-scan coverage is still first-leaf only and ignores sort-only scans.** The dynamic-index spec requires `planner.full_scan` warnings for uncovered predicates, sorts, or text searches (`forge/spec/dynamic-indexes.md:139`). `IndexManager::plan` still only considers text search or the first filter leaf, and returns no warning for a bare `order_by` (`forge/crates/storage/src/index.rs:587`, `forge/crates/storage/src/index.rs:596`, `forge/crates/storage/src/index.rs:599`). `first_indexable_leaf` also uses `find_map` across both `AND` and `OR` (`forge/crates/storage/src/index.rs:761`), so a mixed indexed/unindexed filter can report `uses_index = true` with no warning while part of the query scans. Require every relevant branch/sort key to be covered before suppressing warnings.
