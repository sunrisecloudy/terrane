# Commit Review: b8d12b5e storage dynamic indexes

Reviewed commit:
- `b8d12b5e forge-storage: fix query stable field_id addressing + wire dynamic-index engine (DL-5/DL-6/DL-16)`

## Findings

1. **P1 - Active FTS indexes go stale after any write.** The dynamic-index spec requires inserts, updates, and deletes to refresh the FTS row in the same logical write transaction when the index is active (`forge/spec/dynamic-indexes.md:83`). This commit only populates FTS during `rebuild_active` (`forge/crates/storage/src/index.rs:336`, `forge/crates/storage/src/index.rs:367`), while normal projection writes still only upsert `records` (`forge/crates/storage/src/lib.rs:338`) and mutation/update/delete paths route through that same projection write (`forge/crates/storage/src/lib.rs:627`, `forge/crates/storage/src/lib.rs:646`, `forge/crates/storage/src/lib.rs:667`, `forge/crates/storage/src/lib.rs:686`). Since active text search reads the shadow table directly (`forge/crates/storage/src/lib.rs:545`), a matching record inserted or patched after `build_indexes()` is invisible until a manual rebuild. Add active-index maintenance or mark the index stale on every affected write, with a test that builds FTS, writes, then queries without rebuilding.

2. **P1 - `text_search` bypasses the rest of the query semantics.** DL-15 defines text search as part of the same query surface as filter, sort, limit/offset, aggregates, and group (`prd-merged/02-data-layer-prd.md:68`). Here `query_planned` returns immediately for `q.text_search` (`forge/crates/storage/src/lib.rs:513`), and `run_text_search` only gathers FTS/fallback ids and envelopes (`forge/crates/storage/src/lib.rs:545`, `forge/crates/storage/src/lib.rs:555`); it never applies `q.filter`, never returns aggregate/group results, and skips `limit`/`offset` whenever the order is default/rank because `finalize_rows` is only called in the non-rank branch (`forge/crates/storage/src/lib.rs:569`). It also discards FTS rank by sorting the rank path by id (`forge/crates/storage/src/lib.rs:574`) even though `fts_match` already orders by rank (`forge/crates/storage/src/index.rs:410`) and the fixture requests `rank` ordering (`forge/fixtures/indexes/fts_text_search_uses_shadow_table.json:124`). Compose text search as a match source inside the normal query pipeline, then apply the same filter/order/limit/group/aggregate logic.

3. **P2 - Planner coverage ignores sort-only scans and mixed filters.** The spec requires a `planner.full_scan` warning when a predicate, sort, or text search is not covered by an active index (`forge/spec/dynamic-indexes.md:139`). `IndexManager::plan` only checks text search or the first filter leaf, and returns no warning for a bare `order_by` (`forge/crates/storage/src/index.rs:438`, `forge/crates/storage/src/index.rs:447`, `forge/crates/storage/src/index.rs:450`). The first-leaf helper also descends through `AND` and `OR` by `find_map` (`forge/crates/storage/src/index.rs:586`), so `indexed_field = x OR unindexed_field = y` can report `uses_index = true` with no warning even though one branch still scans. Plan order-only queries and require every relevant predicate branch to be covered before suppressing full-scan warnings.

4. **P2 - A field cannot have both expression and FTS indexes.** The dynamic-index definition hash includes `kind` (`forge/spec/dynamic-indexes.md:31`), and DL-5 distinguishes expression indexes for indexed fields from FTS5 shadow tables for full-text fields (`prd-merged/02-data-layer-prd.md:50`). `IndexManager` keys definitions only by `(collection, field_id)` (`forge/crates/storage/src/index.rs:270`), and `register` overwrites any existing definition for that pair (`forge/crates/storage/src/index.rs:288`). A text field that needs equality/order and full-text search cannot represent both physical structures; whichever kind registers last wins. Key definitions by `(collection, field_id, kind)` or otherwise allow both.

5. **P2 - Schema-valid field ids with dots address the wrong JSON path.** Schema mints field ids directly from `ActorId` (`forge/crates/schema/src/collection.rs:154`), and registry validation only rejects empty actor ids (`forge/crates/schema/src/registry.rs:200`). Storage validation allows `.` in identifiers (`forge/crates/storage/src/query.rs:240`), then emits unquoted JSON1 dot paths such as `$.field_ids.<id>` for queries and DDL (`forge/crates/storage/src/query.rs:134`, `forge/crates/storage/src/index.rs:212`). A valid actor id like `dev.01` produces `f_dev.01_0`, which JSON1 treats as nested keys rather than the literal stable field id. Either constrain schema-minted field ids to path-safe tokens or emit quoted JSON paths like `$."field_ids"."f_dev.01_0"` consistently in query and index DDL.
