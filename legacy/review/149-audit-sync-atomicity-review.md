## Review: d1c4f0aa audit live wiring

Found one transaction-boundary issue in the new sync-RBAC audit persistence.

- **P1 - Sync audit rows can fail after the authorized import already committed.** `WorkspaceCore::sync_with` collects sync audit rows during authorization, but persists them only after `forge_sync::sync_stores_authorized` returns (`forge/crates/core/src/workspace.rs:537`, `forge/crates/core/src/workspace.rs:583`). At that point the receiving stores have already run `apply_remote_chunks`, whose chunk/oplog/projection/schema writes commit in their own transaction (`forge/crates/sync/src/lib.rs:674`, `forge/crates/storage/src/crdt_write/remote.rs:51`). If `persist_audit_rows` then fails (corrupt/overflowed audit counter, disk-full, schema error, etc.), `sync_with` returns `Err` while the records/schema were already imported and no durable audit row exists. That violates the new audit contract that a decision and its row "commit — or roll back — coherently" and that a committed decision always lands its row (`forge/spec/audit-log.md:55`). Please move the audit append into the same receiving-store transaction as the allowed import, or make the sync apply rollback/abort before import whenever the audit rows cannot be reserved/appended; add a fault-injection test where audit append fails and assert no remote chunks/projection changes persist.
