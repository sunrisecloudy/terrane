# Review: `a06cabf8` workspace export/import

Findings for Claude:

- **P1: Export/import copies are not atomic or snapshot-consistent.** `write_bundle` copies each source table with separate autocommit `SELECT` statements and separate destination inserts (`forge/crates/storage/src/export.rs:203-215`), so a concurrent writer can make the bundle contain `oplog`, `crdt_chunks`, and `records` from different moments. Import has the same partial-write shape: it creates the target file, streams tables, then rebuilds (`export.rs:231-245`, `export.rs:263-275`). If any copy/rebuild fails, a partial target DB remains and the next retry is refused because the file now exists. `forge/spec/workspace-export-format.md:7` says import copies/migrates into the target inside a transaction. Use a source read transaction/snapshot plus a destination transaction, and import via temp target + atomic rename or cleanup-on-error.

- **P1: Import can silently drop records that are present in the exported bundle.** Export copies `records` (`export.rs:211`, `export.rs:511-535`), but import intentionally ignores those rows and rebuilds only from `crdt_chunks` (`export.rs:268-275`; `forge/crates/storage/src/crdt_write.rs:456-488`). Any record written through the still-public projection APIs (`forge/crates/storage/src/lib.rs:450-468`, plus legacy/query-fixture paths) is included in the bundle and then lost after import, violating DL-24's byte-identical projection requirement (`prd-merged/02-data-layer-prd.md:83`). Either fail export/import when projection rows lack CRDT history, migrate them into chunks before export, or preserve/adjudicate copied `records` instead of dropping them silently.

- **P1: Snapshot-only CRDT history will not rebuild after import.** `crdt_snapshots` are copied (`export.rs:476-506`) but `Store::rebuild_projection()` only discovers docs from `crdt_chunks` and calls `load_doc_tx`, which also reads only chunks (`forge/crates/storage/src/crdt_write.rs:154-166`, `forge/crates/storage/src/crdt_write.rs:456-488`). Once DL-19 compaction folds old chunks into snapshots (`prd-merged/02-data-layer-prd.md:75`), importing a valid compacted bundle would lose pre-snapshot state. Either teach rebuild/import to seed from `crdt_snapshots` plus later chunks, or mark snapshots as non-authoritative/deferred and do not claim snapshot-safe DL-24 round trips yet.

- **P2: The bundle is not self-contained for index portability.** `import_workspace` requires an external `IndexManager` (`export.rs:231-245`), and the tests reuse the source manager (`export.rs:756-790`), so the imported file alone cannot rebuild query indexes on another device. DL-24 names index defs as part of the workspace export (`prd-merged/02-data-layer-prd.md:83`), and the fixture still lists `index_defs` as missing for GA. Until index definitions are persisted in the bundle, either document this as M0a-only and do not claim self-contained index portability, or include enough index metadata to rebuild with an empty manager.

- **P2: The local-only namespace guard misses exact root buckets.** `is_local_only_namespace` excludes `secret/`, `provider/`, `credentials/`, `__device/`, and `__local/`, but the exact namespace set only includes `secrets`, `device`, and `local` (`export.rs:140-154`). Rows under exact namespaces like `secret`, `provider`, `credentials`, `__device`, or `__local` would be exported even though the prefix comments say those buckets are secret/device-local. Add the exact bucket names and tests for them.
